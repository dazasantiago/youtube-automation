# Signal Scraper — Architecture Reference

> Single source of truth for this codebase. Read before making changes.

---

## What this system does

A daily Python pipeline runs on GitHub Actions. It pulls raw signals from 8 web sources and scans ~50 YouTube competitor channels. All scraped data lands in `data/intel.db` (SQLite) and is exported to `data/signals.json` — a structured JSON file intended to be fed to an LLM for topic classification and editorial decisions.

**The pipeline does not evaluate, cluster, score, or categorize signals.** All analysis happens downstream (outside this repo).

---

## Architecture

```
GitHub Actions (3 daily crons) → run_pipeline.py --stage <name>
│
├── discovery_morning (11:00 UTC)   HN · Reddit · RSS · HF · GitHub · PH · X · GTrends
├── youtube_scan      (12:00 UTC)   ~50 channels: new uploads, median refresh, outlier detection
└── discovery_afternoon (21:00 UTC) HN · Reddit · X (lighter re-scan)
         │
         ▼
    data/intel.db  (SQLite — committed to repo)
         │
         ▼
    data/signals.json  (JSON export, last 3 days, grouped by source)
```

## Database schema

| Table | Purpose | Key constraint |
|---|---|---|
| `signals` | Raw items from all sources | `UNIQUE(source, source_id)` — deduplicates across runs |
| `yt_channels` | ~50 competitor channels | `uploads_playlist_id` cached — never re-fetch |
| `yt_videos` | Competitor uploads with view counts | `outlier_ratio` = views / channel_median |
| `quota_log` | YouTube API quota tracking | Daily sum must stay ≤ 7,500 units |
| `run_log` | Pipeline run metadata | — |

The DB is committed to the repo after every run. Marked binary in `.gitattributes`.

## Source adapters

Each adapter extends `SourceAdapter` and returns a list of `RawSignal` objects:

```python
@dataclass
class RawSignal:
    source: str
    source_id: str
    title: str
    url: str | None
    description: str | None
    posted_at: datetime
    raw_metrics: dict[str, object]
    language: str | None
```

`ingest.py` calls each adapter, then `INSERT OR IGNORE INTO signals` (deduplicated by `UNIQUE(source, source_id)`).

## Output: `data/signals.json`

Generated by `export.py` after `discovery_morning` (or on demand via `content-intel export`).

Structure:
```json
{
  "generated_at": "2026-06-18T...",
  "window_days": 3,
  "total_signals": 412,
  "signals_by_source": {
    "hn":     [{"id": "...", "title": "...", "url": "...", "description": "...", "posted_at": "...", "language": "...", "metrics": {...}}, ...],
    "reddit": [...],
    "rss":    [...],
    ...
  },
  "yt_competitor_videos": [
    {"video_id": "...", "title": "...", "url": "...", "published_at": "...", "views": 12000, "outlier_ratio": 4.2, "language": "en", "channel": "..."},
    ...
  ]
}
```

Feed this file to an LLM to organize signals by topic and identify trends.

## YouTube quota

Hard cap: **7,500 units/day**. Every YouTube API call is logged in `quota_log`. The `run_yt_scan()` function checks the daily total before each call and halts if exceeded.

- `channels.list` (bootstrapping only) — 1 unit/call
- `playlistItems.list` (new uploads) — 1 unit/call per channel page
- `videos.list` (stats refresh) — 1 unit/call per 50 videos

## Gotchas

- **Windows WAL lock:** Always `conn.close()` explicitly; never leave connections open across temp cleanup.
- **uv trampoline (Windows):** Use `uv run python -m pytest`, not `uv run pytest`.
- **YouTube `uploads_playlist_id`:** Permanent per channel — cache once, never re-fetch.
- **Bot commit race:** DB committed after every run; `git pull --rebase --autostash` before push.
